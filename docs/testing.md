# Testing Matrix

Long-form companion to the **Testing** section of `AGENTS.md`. Describes what gets tested where, how to write tests, and what to do about Android-specific coverage under AGP 9.0's KMP limitations.

## The matrix

| Layer | Home source set | Owner module | Example target |
|---|---|---|---|
| Pure logic (reducers, mappers, validators) | `commonTest` | every module that has logic | `:core:settings`, `:feature:home:impl` |
| ViewModel with fake deps | `commonTest` of feature `:impl` | `:feature:*:impl` | `HomeViewModelTest` |
| Repository with in-memory Room | `commonTest` of `:data:*` | `:data:*` | `SampleRepositoryTest` (when repo exists) |
| Platform actuals (DB builder, Settings impl, Ktor engine wiring) | `androidMain`/`iosMain` tests | the module that owns the `actual` | `:core:settings`, `:composeApp` |
| Compose UI snapshots | `androidUnitTest` + Roborazzi | module whose UI is under test | `:core:ui`, future `:feature:*:impl` |
| Integration / end-to-end UI | `androidInstrumentedTest` (future) | `:composeApp` | happy-path flows |
| Performance | `:composeApp/benchmarks` (future) | `:composeApp` | cold start, scroll, nav |

One column per source set; one row per behaviour class. If a test doesn't fit any row, ask whether it's a real test or a missing production abstraction.

## commonTest — pure and VM

Every module has a `commonTest` source set. Common deps (Kotlin test, coroutines-test, Turbine, multiplatform-settings-test) are added per-module as needed.

```kotlin
class HomeViewModelTest {

    @Test
    fun addItem_emitsEffect_andClearsInput() = runTest {
        val repository = FakeSampleRepository()
        val vm = HomeViewModel(repository)

        vm.state.test {
            assertEquals(HomeUiState(), awaitItem())
            vm.onTitleChanged("t")
            vm.onDescriptionChanged("d")
            awaitItem() // title edit
            awaitItem() // description edit
            vm.onAddClicked()
            val saving = awaitItem()
            assertTrue(saving.isSaving)
            val saved = awaitItem()
            assertEquals("", saved.inputTitle)
            cancelAndIgnoreRemainingEvents()
        }
    }
}
```

- **Fakes over mocks.** Mockito/mockk are JVM-only and off-limits in `commonTest`. Write a hand-rolled `FakeSampleRepository` that implements the same interface.
- **`runTest` + `UnconfinedTestDispatcher`** for deterministic state-flow ordering.
- **Turbine** for `Flow<T>` assertions. `state.test { awaitItem() ... }` consumes emissions in order. `cancelAndIgnoreRemainingEvents()` after the last asserted emission.
- No `Thread.sleep` / `delay`. If a test needs time to pass, use `advanceTimeBy` on the test scheduler.

## Platform actuals — androidMain / iosMain tests

Each `actual` implementation deserves a test in its own source set:

- `:core:settings/src/androidUnitTest/...` — `SharedPreferencesSettings` round-trip (needs Robolectric).
- `:composeApp/src/iosTest/...` — `getDatabaseBuilder()` iOS path construction.

iOS tests run via `:core:settings:iosSimulatorArm64Test` etc. Android unit tests are blocked on AGP 9.0 (see caveat below).

## Compose UI snapshots (Roborazzi)

- Wired today on `:core:ui` for `AppTheme` coverage. Golden PNGs live in `core/ui/src/androidUnitTest/snapshots/`.
- Each feature `:impl` module may opt in: add the `roborazzi` plugin + `androidUnitTest.dependencies` block (mirror `:core:ui`), then write `@Test` cases that call `composeRule.setContent { Screen(...) }` + `composeRule.onRoot().captureRoboImage()`.
- **Review PNG diffs manually before committing.** A green `verifyRoborazziDebug` without visual review defeats the point.

**AGP 9.0 caveat:** `com.android.kotlin.multiplatform.library` does not yet expose an Android unit test Kotlin compilation, so `recordRoborazziDebug` / `verifyRoborazziDebug` are not yet registered. The test source files are preserved for when support lands.

## Integration + instrumentation

Reserved for `:composeApp` once:
1. AGP unblocks Android unit tests on KMP libraries, or
2. A dedicated Android app module hosts the instrumentation suite.

Until then, happy-path flows are verified manually on an Android device and iOS simulator before each release.

## Performance

See `docs/performance.md`. Macrobenchmark runs from a future `benchmarks` module against `:androidApp`. Baseline Profiles are generated by a profile-generator module that exercises top user journeys.

## Execution

```bash
./gradlew composeApp:allTests              # common + ios tests in composeApp
./gradlew core:settings:allTests           # per-module
./gradlew check                            # everything
```

`./gradlew check` depends on all `:<module>:allTests` and detekt; CI runs it on every PR.
