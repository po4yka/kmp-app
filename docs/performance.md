# Performance Budgets

Long-form companion to the **Performance** section of `AGENTS.md`. Budget categories, tooling, and what gets measured when.

## Budget categories

Starter budgets for this template measured on a mid-tier Android reference device (Pixel 6a class, Android 14) and an iPhone 13 / iOS 17 reference. They are deliberately generous — tighten them against the first release-candidate measurement, never regress afterwards.

| Category | Metric | Reference device | Starter target | Who measures |
|---|---|---|---|---|
| Cold start (no Baseline Profile) | TTID (time-to-initial-display) | Pixel 6a | ≤ 800 ms median | Macrobenchmark, `StartupTimingMetric` |
| Cold start (with Baseline Profile) | TTID | Pixel 6a | ≤ 600 ms median | Macrobenchmark, `StartupTimingMetric` |
| Warm start | TTID | Pixel 6a | ≤ 300 ms median | Macrobenchmark |
| First frame after route change | ms from `backStack.add(...)` → first frame | Pixel 6a | ≤ 200 ms median | Macrobenchmark, `FrameTimingMetric` |
| Scroll p95 | Frame time 95th percentile on `LazyColumn` (Home screen, 100 items) | Pixel 6a | ≤ 16 ms | Macrobenchmark, `FrameTimingMetric` |
| Jank rate | Frames > 33 ms during scroll | Pixel 6a | ≤ 1% | Macrobenchmark, `FrameTimingMetric` |
| Memory peak | Max RSS during a representative session (Home + Detail + add 20 items) | Pixel 6a | ≤ 120 MB | `adb shell dumpsys meminfo`, Android Studio Profiler |
| APK size (release, minified, no Baseline Profile) | Install size | — | ≤ 15 MB | CI artifact report |
| APK size (release, minified, with Baseline Profile) | Install size | — | ≤ 17 MB | CI artifact report |
| IPA size (release, arm64) | Install size | — | ≤ 50 MB (Compose Multiplatform static framework is chunky by nature) | CI artifact report |
| iOS cold start | Time from icon tap → first frame | iPhone 13 | ≤ 700 ms median | Xcode Instruments (*App Launch*) |

The RC1 pass sets the authoritative numbers per device tier. Budget is "don't regress from RC1 + 5%". Every PR that changes a hot path runs the suite; anything beyond the budget fails CI.

### Notes on the starter numbers

- Baseline Profile typically shaves ~30% off first-launch execution on Android; the two cold-start rows reflect that.
- 16 ms scroll p95 is the 60 fps floor. Higher-refresh-rate devices (90–120 Hz) are treated as bonus headroom, not a new budget — displaying smoothly at 60 fps on a mid-tier device is the floor.
- iOS APK size is larger because the `ComposeApp` static framework includes the Kotlin runtime, Compose Multiplatform binaries, and every KMP klib statically linked. Dynamic frameworks would shrink it but add launch cost.

## Tooling

### Baseline Profile

Baseline Profiles precompile hot code paths for the first launch. They typically improve first-launch execution by ~30% on Android. Generated by exercising top user journeys in a profile-generator test module.

```bash
./gradlew :androidApp:generateBaselineProfile
```

Generated profile lands in `androidApp/src/release/generated/baselineProfiles/baseline-prof.txt`. Commit it.

### Startup Profile

Startup Profiles optimize DEX file layout specifically for startup. Generated the same way; output drops in `startup-prof.txt`. Combined with a baseline profile, they produce measurable cold-start improvements on the first launch after install.

### Macrobenchmark

Tracks metrics across user journeys from an app-under-test standpoint. Each benchmark measures one journey N times and reports median + p95.

Top-3 journeys for this app (wired once the first RC is built):
1. Cold start → Home screen visible.
2. Home → tap item → Detail screen visible.
3. Home → add item → row appears in list.

Reports land in Android Studio's Profiler view plus a machine-readable JSON for CI trend tracking.

### Compose compiler stability

Composables skip recomposition when their inputs are stable (primitive or `@Immutable`). Use the Compose compiler's stability report to find unstable params:

```bash
./gradlew :composeApp:compileDebugKotlinAndroid -PcomposeCompilerReports=build/compose_compiler
```

Inspect `build/compose_compiler/*-classes.txt` for `unstable` flags on shared data classes. Fix by annotating with `@Immutable` (for data classes whose fields are all stable) or by making collections read-only types.

## Compose stability checklist

- Every data class emitted by a VM (`UiState`, sub-state) is annotated `@Immutable` if Compose can't prove stability.
- Collections in state are `List<T>` / `Set<T>` / `Map<K, V>` (immutable types) — not `MutableList`, not `ArrayList`.
- Lambda captures do not reference unstable state across composable boundaries.
- No `currentTimeMillis()` / `Random.nextX()` inside a composable body.
- `key = { ... }` on `items(...)` to avoid unnecessary item recomposition.
- Heavy composables go behind `derivedStateOf` when they read state transitively.

## Process

1. First RC: run the full Macrobenchmark suite, pin numbers in the table above.
2. PR-time: CI runs Macrobenchmark on the most-affected journeys. Regressions > 5% of target fail the build.
3. Release-time: regenerate Baseline + Startup profiles, commit them.
4. Quarterly: re-baseline budget numbers against new device baselines (mid-tier Android hardware evolves).
